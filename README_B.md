# Phase B — Resume Parse Slice

> Part of the [AI Job Assistant](README.md) phased build. Phase B is the first real
> AI slice — see [Phase A](README_A.md) for the walking skeleton it builds on.

## Goal

Turn an uploaded resume PDF into a validated, persisted `ParsedResume`, fully traced,
with a parse-quality eval harness enforcing an accuracy bar and a zero-fabrication gate.

## What was built

| Piece | Where | Notes |
|---|---|---|
| Output schema | `app/schemas.py` | `ParsedResume` + nested `WorkEntry`, `EducationEntry`; every field nullable/empty-default (first fabrication guardrail) |
| DB models | `app/models.py` | `Resume` (source file) + `ParseRun` (one row per parse attempt, JSONB `parsed`); one-to-many |
| Migrations | `migrations/` (Alembic, async) | `resume` + `parse_run` tables, autogenerated from the models |
| PDF ingestion | `app/parse.py` `extract_text()` | `pypdf`, page-by-page (ADR-0012) |
| LLM parse | `app/parse.py` `parse_resume()` | OpenAI Structured Outputs, `temperature=0`, no-fabrication system prompt, versioned (`PROMPT_VERSION`) |
| DB access | `app/db.py` | lazy async SQLAlchemy engine/session (same lazy-config pattern as Phase A) |
| Upload endpoint | `app/main.py` `POST /resumes` | validate PDF → persist `Resume` → extract+parse → persist `ParseRun` (success **or** failure, both recorded) → return result |
| Eval harness | `tests/eval/eval_parse.py` + `fixtures/` | 3 synthetic fixture pairs (`.txt` + `_expected.json`); scores field accuracy, skills precision/recall, per-entry work/education matching, zero-fabrication gate |

**Observability:** every parse call is traced via the same Langfuse `openai` drop-in from
Phase A; `parse-resume` traces show up per call.

## Key decisions

- **ADR-0012 — `pypdf` for ingestion.** Local, deterministic, in-process text extraction;
  OCR/vision-model fallback deferred, not needed for text-based resumes.
- **ADR-0015 — reject RAG-augmented parsing.** A "B2" extension (Chroma-based dual-purpose
  chunking, for both extraction context and semantic fabrication-checking) was designed
  and then **explicitly rejected** — Phase B ships as the base slice only, with strict
  string-match evaluation. See the ADR for the full reasoning.
- **`temperature=0`.** The parse call originally had no temperature set, and the eval
  caught real run-to-run flakiness (an identical fixture scored differently between two
  runs). Pinning `temperature=0` made extraction deterministic — confirmed by running the
  eval twice back-to-back with identical results.

## In depth

### The eval caught two real bugs before they shipped

This phase is a good example of "evals are the new unit tests" paying off immediately,
not just in theory:

1. **CI lint gap.** Running the *actual* CI command (`ruff check .`, the whole repo) for
   the first time surfaced that Alembic's autogenerated migration files didn't pass lint
   — never caught because earlier checks only linted `app/` and `tests/eval/`
   individually. Fixed by excluding `migrations/versions/` (autogenerated, not
   hand-styled) from ruff, while keeping `migrations/env.py` (hand-edited) linted.
2. **Non-deterministic extraction.** The eval harness re-run on an unchanged fixture
   produced a different score (2/2 → 0/2 on one education entry) purely from OpenAI's
   default sampling temperature. Since a pass/fail gate that flips on identical input is
   useless, this was fixed by pinning `temperature=0` — verified stable across repeated
   runs afterward.

### Why the schema is the load-bearing artifact

`ParsedResume` (and nested `WorkEntry`/`EducationEntry`) does **triple duty**: it's the
Structured Outputs contract, the JSONB persistence payload, and the eval's scoring
target. The key discipline is nullable-with-default on every field (`str | None = None`,
lists default `[]`) — that's the fabrication guardrail encoded directly in types: a
non-nullable field would force the model to invent something even when the resume states
nothing.

### Why B2 (RAG-augmented parsing) was rejected

An earlier design session floated chunking + embedding resume text into Chroma, used
both to give the LLM retrieval-augmented context and to semantically (cosine-similarity)
verify extracted skills aren't fabricated — arguably more robust against paraphrase
("Docker" vs. "containerization") than strict string matching. It was designed, then
explicitly decided against (ADR-0015): the single-resume, single-call extraction task
doesn't need retrieval (the whole source text already fits in the prompt), and the added
vector-store infrastructure and second eval pathway weren't judged worth it for this
phase. If real-world paraphrase false-positives become a real problem, that's a new ADR,
not a reopening of this one.

## Running it locally

Prereqs: same as Phase A (`uv`, Docker, OpenAI key, Langfuse), plus the app Postgres
migrated.

```bash
# 1. App database (if not already running)
docker compose up -d

# 2. Apply migrations (creates `resume` + `parse_run` tables)
uv run alembic upgrade head

# 3. Run the app
uv run uvicorn app.main:app --env-file .env --reload
```

Try it — either the interactive docs (`http://localhost:8000/docs` → `POST /resumes` →
upload a PDF), or curl:

```bash
curl -X POST http://localhost:8000/resumes \
  -F "file=@/path/to/resume.pdf;type=application/pdf"
```

Verify:

```bash
docker compose exec db psql -U app -d app -c "SELECT id, filename FROM resume;"
docker compose exec db psql -U app -d app -c "SELECT id, status, model FROM parse_run;"
# Langfuse UI (http://localhost:3000) -> Traces -> `parse-resume`
```

Run the eval harness (real OpenAI calls, not part of CI):

```bash
uv run --env-file .env python -m tests.eval.eval_parse
```

Lint + test:

```bash
uv run ruff check .
uv run pytest tests/
```

## Exit criteria

| Criterion | Status |
|---|---|
| PDF in → persisted `ParsedResume` out | ✅ |
| Visible as a Langfuse trace | ✅ |
| Parse eval reports field accuracy + fabrication count | ✅ |
| ≥95% accuracy, 0 fabrication | ✅ (100% / 0, across 3 fixtures, stable across repeated runs) |
| CI green (`ruff check .` on the whole repo) | ✅ |

## Notes

- Model is still **`gpt-4o-mini`** as a placeholder (`app/parse.py` `MODEL`) — pinning
  after a feasibility spike remains an open, non-blocking item.
- `prompt_version` (`PROMPT_VERSION = "parse-v1"`) is recorded on every `ParseRun` row —
  bump it whenever `SYSTEM_PROMPT`'s wording changes, so historical rows stay correctly
  labeled by which prompt actually produced them.
- GitHub issue #5 closed 2026-07-11; see ADR-0015 for the RAG-augmented-parsing decision.

## What comes next — Phase C

Phase C closes the MVP loop: a **LangGraph `interrupt()`** human-review gate (the
applicant sees the parse and approves or corrects it) plus a minimal separate front end
driving upload → review → approve. ADR-0011 (LangGraph checkpointer) is still open and
blocks that phase's persistence design.
